1. Три слоя (ментальная модель)

main — старт, middleware, роуты → handler — HTTP: разобрать запрос, вызвать данные, ответ → store — БД: SQL, bcrypt, без HTTP
Lab (Go)Spring (корп, позже)Зона ответственности
cmd/server/main.goApplication + конфигСобрать приложение
internal/handler/*.go@RestControllerHTTP-контракт
internal/store/*.go@RepositoryДанные, SQL
(нет в lab)@ServiceБизнес-логика между controller и repo
В lab слой service намеренно пропущен — handler сразу зовёт store. В проде сложную логику выносят в service. На собесе: «В lab два слоя, в Spring — три, роли те же».

2. Разбор по файлам: GET /notes

① main.go — кто принимает порт

st, _ := store.Open(cfg.DatabasePath)     // один Store на всё приложение
api := app.Group("/api/v1")
handler.NewNotes(cfg, st).Register(api)   // передаём store в handler

Создаётся один store (подключение к SQLite), передаётся в handlers через конструктор NewNotes(cfg, st) — это dependency injection вручную (в Spring — автоматически).

② notes.go — Register и middleware на группе

notes := router.Group("/notes", middleware.IsAuthed(h.cfg))
notes.Get("/", h.list)

Все /notes/* сначала проходят IsAuthed (Урок 0002). Потом — метод list.

③ list — handler тонкий

func (h *NotesHandler) list(c *fiber.Ctx) error {
  list, err := h.store.ListNotes(middleware.UserID(c))
  if err != nil { return response.Error(c, 500, "list failed") }
  return response.JSON(c, 200, "ok", fiber.Map{"list": list})
}

Handler не пишет SQL. Он: достал userId из context (положил middleware) → позвал store → обернул в envelope. Это хороший handler.

④ store.go — SQL

func (s *Store) ListNotes(userID uint) ([]model.Note, error) {
  rows, err := s.db.Query(
    `SELECT id, user_id, title, body, created_at, updated_at
     FROM notes WHERE user_id = ? ORDER BY id DESC`, userID)
  // ... scan rows в слайс
}

Store не знает про HTTP, JWT, JSON. Только: «дай заметки этого user_id». Фильтрация по пользователю — в SQL, не на фронте (иначе утечка чужих данных).
