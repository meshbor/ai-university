Сегодняшняя победа. Пройдёшь POST /api/v1/auth/login по строкам кода — от JSON в теле до Set-Cookie — и скажешь, какой слой за что отвечает. Это тот же паттерн, что в Spring Security + UserRepository.

1. Карта файлов

handler/auth.go     — login(), Register(), cookie, ответ
store/store.go      — FindUserByEmail, CheckPassword (bcrypt)
pkg/jwt/jwt.go      — Sign(userId, role, ttl)
pkg/validate        — правила email/password
middleware/auth.go  — НЕ на login (токен ещё нет)

2. login() — семь шагов

Держи открытым internal/handler/auth.go и сверяйся.

Шаг 1 — HTTP → struct. BodyParser кладёт JSON в loginPayload{Email, Password}. Ошибка парсинга → 400 invalid json.
Шаг 2 — валидация. validate.Struct(body): email обязателен, пароль ≥ 6. Ошибка → 400 (не 401!). См. Урок 0007.
Шаг 3 — найти юзера. h.store.FindUserByEmail(body.Email) → SQL SELECT ... FROM users WHERE email = ?. Нет строки → ErrNotFound.
Шаг 4 — проверить пароль. h.store.CheckPassword(user, body.Password) → bcrypt.CompareHashAndPassword с хэшем из БД. В БД пароль никогда открытым текстом (см. seed в store).
Шаг 5 — если шаг 3 или 4 не ок. Один ответ: 401 invalid credentials (не раскрываем «нет такого email» vs «пароль неверный» — безопасность).
Шаг 6 — выдать JWT. h.jwt.Sign(user.ID, user.Role, 8*time.Hour) → строка токена. Подпись секретом JWT_SECRET из config. Ошибка → 500.
Шаг 7 — ответ клиенту. Cookie access_token (HttpOnly, 8ч) + response.JSON с data.user и data.accessToken (дубль для curl/мобилки).
Почему handler не пишет SQL и не вызывает bcrypt напрямую? Те же причины, что в Уроке 0006: handler = HTTP-оркестратор; store = всё про данные. В Java: AuthController зовёт UserRepository, не пишет SELECT сам.
