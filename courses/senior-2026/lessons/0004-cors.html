Урок 0004 · CORS и credentials

Курс «Senior 2026» · Фаза 0 · ~15 мин · fullstack-lab (CORS в main.go + proxy Vite)

Сегодняшняя победа. Поймёшь, почему браузер иногда блокирует запрос до того, как он дойдёт до сервера; чем CORS-ошибка отличается от 401; зачем credentials: 'include' и preflight OPTIONS; и почему UI lab на :5173 «магически» работает без CORS-ада.

1. Теория (коротко)

Same-origin и зачем CORS

Origin = схема + хост + порт. http://localhost:5173 и http://localhost:8080 — разные origin (порты разные). Браузер по умолчанию не даёт JS на странице с :5173 читать ответ с :8080 — это same-origin policy.

CORS (Cross-Origin Resource Sharing) — механизм, при котором сервер явно разрешает другому origin читать ответ через заголовки вроде Access-Control-Allow-Origin.

CORS ≠ 401

СитуацияГде видишьСмысл
CORS blockКрасная ошибка в Console, в Network запрос может быть «failed»Браузер запретил JS прочитать ответ. До бизнес-логики может дойти, но фронт ответ не увидит.
401HTTP 401, JSON {"data":{},"status":401,...}Запрос дошёл до API, middleware отклонил — нет/битый токен.

На собесе: «CORS — защита браузера; 401 — решение приложения».

CORS_ORIGINS на бэке — что это

В config.go по умолчанию CORS_ORIGINS = http://localhost:5173. Это whitelist: «браузеру со страницы именно этого origin разрешено читать ответ API :8080». В main.go Fiber подставляет это в Access-Control-Allow-Origin. Если фронт в проде на https://admin.example.com — добавляешь этот URL в env. * (все origins) нельзя вместе с cookie (AllowCredentials).

credentials в fetch — три режима

ЗначениеCookie / AuthorizationКогда
omitНикогда не шлёт cookie и не принимает Set-Cookie для этого запросаПубличные API без логина; сторонние виджеты
same-origin (дефолт)Cookie только если URL страницы и запроса — один originОбычный fetch на свой же домен
includeШлёт cookie и на другой origin (если CORS разрешил)SPA на :5173 → API на :8080 без proxy; lab явно ставит include

omit — «режим инкогнито для credentials»: даже если cookie есть, fetch их не приложит. same-origin — cookie на /api через Vite proxy (:5173→:5173) работает. include — нужен для cross-origin + cookie (и строгий CORS на бэке).

credentials: 'include' и cookie (lab)

Чтобы браузер слал cookie на другой origin, нужно два условия:

  • на фронте: fetch(..., { credentials: 'include' }) — так в lab;
  • на бэке: Access-Control-Allow-Credentials: true и Allow-Origin = конкретный URL (не *).

В lab (main.go): AllowCredentials: true, origins из CORS_ORIGINS (по умолчанию http://localhost:5173).

Preflight и метод OPTIONS — в чём разница слов

Preflight — не отдельный HTTP-метод, а название шага в спецификации CORS: «предполётная проверка» перед «настоящим» запросом. Браузер реализует preflight запросом с методом OPTIONS.

Термин (что происходит)     HTTP-метод (как в wire)
─────────────────────────   ───────────────────────
preflight-запрос        →   OPTIONS  + заголовки Origin, Access-Control-Request-*
основной запрос         →   POST / GET / … (тот, что ты хотел)

Аналогия: «досмотр в аэропорту» (preflight) — процедура; «показать паспорт» (OPTIONS) — конкретное действие. На собесе можно сказать: «Preflight — это OPTIONS от браузера перед небезопасным cross-origin запросом».

curl preflight не делает — только браузер. В практике ты имитируешь его вручную: curl -X OPTIONS ...

Почему UI lab «не ломается» на CORS

Vite проксирует API:

// apps/web/vite.config.ts
proxy: { '/api': { target: 'http://localhost:8080' } }

Браузер видит запрос на http://localhost:5173/api/v1/notes — тот же origin, что и страница. Vite на сервере пересылает на :8080. CORS для UI в dev не нужен. CORS на API всё равно настроен — для прода, когда фронт и API на разных хостах, или если бьёшь в :8080 из консоли браузера.

Связь с yamal. Gateway тоже включает CORS с AllowCredentials — иначе админка на одном домене не прочитает API на другом.

3. Практика на полигоне

# 1. Preflight руками (curl имитирует браузер)
curl -i -X OPTIONS http://localhost:8080/api/v1/notes \
  -H "Origin: http://localhost:5173" \
  -H "Access-Control-Request-Method: POST" \
  -H "Access-Control-Request-Headers: Content-Type, Authorization"

# Ищи в ответе: Access-Control-Allow-Origin, Allow-Credentials, Allow-Methods

# 2. В Chrome на http://localhost:5173 — открой Console (F12) и выполни:

# 2a. Cross-origin напрямую на :8080 (часто CORS error в консоли)
fetch("http://localhost:8080/api/v1/notes", { credentials: "include" })

# 2b. Через proxy Vite (same-origin) — увидишь 401 JSON, не CORS
fetch("/api/v1/notes", { credentials: "include" }).then(r => r.json()).then(console.log)
