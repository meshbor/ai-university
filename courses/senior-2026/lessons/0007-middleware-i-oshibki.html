Урок 0007 · Middleware, валидация и ошибки

Курс «Senior 2026» · Фаза 1 · main.go, middleware/auth.go, validate, response.Error

Сегодняшняя победа. Поймёшь цепочку middleware до handler, как валидируется вход, как handler выбирает статус ошибки — и воспроизведёшь это curl'ом (400, 401, 404).

1. Два вида middleware

Глобальные (на всё приложение) — main.go

запрос → recover → requestid → logger → cors → маршрут
  • recover — паника в handler → 500, сервер не падает.
  • requestid — уникальный id запроса в логах (сквозная трассировка; в yamal важно между сервисами).
  • logger — пишет метод, путь, статус, время.
  • cors — заголовки для браузера (Урок 0004).

На группу маршрутов — IsAuthed

notes := router.Group("/notes", middleware.IsAuthed(h.cfg))

Только для /notes и /auth/me, /logout. POST /auth/login — без auth middleware (иначе замкнутый круг).

2. Валидация входа

type loginPayload struct {
  Email    string `json:"email" validate:"required,email"`
  Password string `json:"password" validate:"required,min=6"`
}
// ...
if err := validate.Struct(body); err != nil {
  return response.Error(c, 400, err.Error())
}

Теги validate: — правила до бизнес-логики. Короткий пароль → 400 Bad Request, не 401. 401 — только когда формат ок, но credentials неверны.

3. Карта ошибок в lab

КодКогда в labПример message
400Битый JSON, валидация, невалидный idinvalid json, Password: min
401Нет/битый токен; неверный логин/парольunauthorized, invalid credentials
404Нет маршрута; нет заметки у этого usernot found, note not found
500Ошибка БД, подписи JWTlist failed, token error

Все через response.Error → тот же envelope {data:{}, status, message}.

400 vs 401 на login. Пароль из 3 символов → 400 (валидация). Пароль 8 символов, но неверный → 401. Различать на ревью и в support-логах.

5. Практика: вызови ошибки curl'ом

# 400 — пароль короче 6 (валидация)
curl -s -X POST http://localhost:8080/api/v1/auth/login \
  -H 'Content-Type: application/json' \
  -d '{"email":"editor@lab.local","password":"abc"}' | python3 -m json.tool

# 401 — формат ок, пароль неверный
curl -s -X POST http://localhost:8080/api/v1/auth/login \
  -H 'Content-Type: application/json' \
  -d '{"email":"editor@lab.local","password":"wrongpass"}' | python3 -m json.tool

# 404 — заметка не существует (сначала login -c jar)
curl -c /tmp/lab.jar -s -X POST http://localhost:8080/api/v1/auth/login \
  -H 'Content-Type: application/json' \
  -d '{"email":"editor@lab.local","password":"lab12345"}' > /dev/null
curl -b /tmp/lab.jar -s http://localhost:8080/api/v1/notes/99999 | python3 -m json.tool

6. Введение: идемпотентность (на будущее)

Идемпотентный метод — повторили запрос N раз → тот же эффект, что один раз. GET — идемпотентен. POST /notes — нет (каждый раз новая заметка). PUT одного id — обычно да. Подробнее в Фазе 3 (API-дизайн). Пока — знай термин.
