Урок 0003 · Контракт API: data, status, message

Курс «Senior 2026» · Фаза 0 · ~15 мин · полигон fullstack-lab (тот же формат, что yamal-media gateway)

Сегодняшняя победа. Ты научишься читать любой ответ lab (и yamal) как единый «конверт», понимать разницу между HTTP-статусом и полем status в теле, и писать на фронте один универсальный обработчик вместо угадывания «а тут JSON другой?».

1. Теория (коротко)

Зачем единый формат

Без договора каждый эндпоинт отдаёт JSON по-своему: где-то массив, где-то {error: "..."}, где-то просто строка. Фронт превращается в кашу из if. Контракт API — это договор: всегда три поля верхнего уровня. Данные — только внутри data.

Конверт (envelope) в lab

{
  "data":    { ... },   // полезная нагрузка (объект, список, пустой {})
  "status":  200,       // дублирует HTTP-статус (см. ниже зачем)
  "message": "ok"       // человекочитаемое пояснение для UI/логов
}

В коде это одна функция — все handlers зовут её, а не собирают JSON руками:

// apps/api/pkg/response/response.go
return ctx.Status(status).JSON(Envelope{
  Data: data, Status: status, Message: message,
})

Успех и ошибка — одна форма

Успех — GET /notes (фрагмент):
{"data":{"list":[...]},"status":200,"message":"ok"}
Ошибка — без cookie:
{"data":{},"status":401,"message":"unauthorized"}

Обрати внимание: при ошибке data — пустой объект {}, не null. Фронт читает json.data только после проверки успеха — см. блок про res.ok ниже.

HTTP-статус и body.status — зачем оба?

  • HTTP-статус (первая строка: HTTP/1.1 401) — для браузера, прокси, CDN, fetch.
  • status в теле — тот же код, но внутри JSON-конверта.

Что за «очереди» и почему дублируем (развёрнуто)

В yamal-media архитектура не «один сервер отвечает сразу». Есть gateway (Go) и микросервисы (Node). Между ними — очередь сообщений RabbitMQ: gateway кладёт задачу в очередь, сервис обрабатывает и кладёт ответ обратно. Это не HTTP — у сообщения в очереди нет строки HTTP/1.1 200. Есть только JSON-тело.

Браузер  --HTTP-->  Gateway  --RabbitMQ-->  Сервис (articles, users…)
                      |                            |
                      <-------- JSON ответ --------<
                      |
              тот же { data, status, message }
              отдаётся браузеру + HTTP 200/401/…

Поэтому в yamal один тип ответа — RmqResponseDTO = { data, status, message }. Поле status в теле говорит сервису и gateway: «успех или ошибка», даже когда HTTP ещё не при чём. Gateway получает JSON из очереди, смотрит body.status и выставляет тот же код в HTTP-строке для браузера. Дублирование намеренное — один DTO на весь стек.

В fullstack-lab очереди нет (упрощение): handler сразу зовёт store. Но формат ответа тот же — чтобы ты учился на том, что увидишь в проде. См. docs/MAPPING-YAMAL.md в lab.

res.ok — что это на фронте

В fetch результат делится на два объекта:

const res = await fetch('/api/v1/notes', { credentials: 'include' })
// res — Response (обёртка HTTP): res.status, res.ok, res.headers
const json = await res.json()
// json — тело: { data, status, message }

res.ok — булево: true, если HTTP-код 200–299; иначе false (401, 404, 500…). В lab-клиенте проверка такая:

if (!res.ok) {
  throw new Error(json.message || `HTTP ${res.status}`)
}
return json  // дальше работаем с json.data

Не путай: res.ok (HTTP) и json.status (поле в теле). В норме совпадают; на фронте обычно сначала res.ok, потом читаем json.data.

Как фронт lab это типизирует

// apps/web/src/api/types.ts
export type ApiEnvelope<T> = {
  data: T
  status: number
  message: string
}

Один generic — и listNotes() знает, что список в data.list, а login() — что юзер в data.user.

4. Практика на полигоне

# Логин + сохранить cookie в jar (как браузер)
curl -c /tmp/lab.jar -s -X POST http://localhost:8080/api/v1/auth/login \
  -H 'Content-Type: application/json' \
  -d '{"email":"editor@lab.local","password":"lab12345"}' | python3 -m json.tool

# Список заметок — только cookie (-b), Bearer НЕ нужен (как UI lab)
curl -b /tmp/lab.jar -s http://localhost:8080/api/v1/notes | python3 -m json.tool

# Ошибка — та же форма конверта; запомни message для автопроверки ниже
curl -s http://localhost:8080/api/v1/notes | python3 -m json.tool

# Создать заметку — вариант 1: только cookie (рекомендуется)
curl -b /tmp/lab.jar -s -X POST http://localhost:8080/api/v1/notes \
  -H 'Content-Type: application/json' \
  -d '{"title":"Контракт API","body":"data status message"}' | python3 -m json.tool

# Увидеть HTTP 201 отдельно (НЕ смешивай -w с json.tool в одной трубе!)
curl -b /tmp/lab.jar -s -o /tmp/note.json -w "HTTP код: %{http_code}\n" \
  -X POST http://localhost:8080/api/v1/notes \
  -H 'Content-Type: application/json' \
  -d '{"title":"Ещё одна","body":"проверка 201"}'
python3 -m json.tool /tmp/note.json

# Вариант 2: Bearer вместо cookie (если сохранил TOKEN из login)
# curl -H "Authorization: Bearer $TOKEN" -s -X POST ...
Про -w "%{http_code}". Флаг -w заставляет curl дописать в конец вывода строку с HTTP-кодом. Если сразу pipe в python3 -m json.tool, парсер JSON ломается — в конце файла не JSON, а HTTP_CODE:201. Решение: сохранить тело в файл (-o), код вывести отдельно (-w), JSON форматировать второй командой.

5. Почему не GraphQL и не «чистый REST» без обёртки

Чистый REST без { data, status, message }

«Чистый» вариант: GET /notes → тело сразу [{...},{...}], ошибка → {"error":"unauthorized"} или просто HTTP 401 без тела. Так делают многие публичные API. Минусы для большой команды / микросервисов:

  • у каждого эндпоинта свой shape ошибки — фронт плодит ветки;
  • ответ из очереди (RabbitMQ) не совпадает с тем, что ждёт фронт — нужны адаптеры;
  • сложнее единый перехватчик (interceptor) в клиенте.

Обёртка envelope — внутренний стандарт: один парсер, одна типизация ApiEnvelope<T>, один логгер. Цена: лишний уровень вложенности (data.list вместо list).

GraphQL

Один URL, клиент сам описывает, какие поля нужны. Плюсы: гибкость для сложных экранов, меньше overfetching. Минусы: другая инфраструктура (схема, резолверы, N+1 на бэке), кэширование HTTP сложнее, learning curve. yamal / lab / твой корп — REST + envelope: проще для gateway + микросервисов + классических админок. GraphQL имеет смысл, когда много разных клиентов с разными наборами полей (мобилка + веб + партнёры).

На собесе: «Мы используем envelope для единого DTO между HTTP и RabbitMQ; GraphQL рассмотрели бы, если бы…» — показываешь, что выбор осознанный, не «потому что так в туториале».
